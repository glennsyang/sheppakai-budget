# Security Code Review - 2026-04-17

## Scope

Full repository static security review with emphasis on SvelteKit server entry points and trust boundaries:

- `+page.server.ts` / `+layout.server.ts`
- `+server.ts`
- `hooks.server.ts`
- Auth/session/password-reset flows
- CRUD action authorization and data-access behavior

**Product requirement update applied:** this app has exactly two users and they intentionally share all application data. Cross-user read access is expected behavior in this product model.

## Executive Summary

The prior finding **"P0: Cross-user data reads from missing userId scoping"** is **not applicable** under the explicit shared-data requirement and has been removed.

Remaining findings focus on admin action authorization, reset-link origin trust, cron-secret fail-closed behavior, and CSP hardening.

## Findings

### [P1] Admin action handlers lack explicit per-action admin enforcement

- **Affected files/functions:**
  - `src/routes/(app)/admin/users/+page.server.ts` (all actions)
  - `src/routes/(app)/admin/archived-goals/+page.server.ts` (`unarchive`)
- **Vulnerability type:** Broken function-level authorization
- **Risk/exploit scenario:** If actions are posted directly, non-admin authenticated users may reach privileged handlers.
- **Evidence pattern:** Actions check `locals.user` but do not always assert admin role within each action function.
- **Recommended fix:** Add explicit `requireAdmin`/role check inside each admin action before executing privileged logic.
- **Verification steps:**
  1. Authenticate as non-admin.
  2. POST directly to admin action endpoints.
  3. Confirm consistent 403 responses.

### [P1] Password reset callback URL built from request origin (origin trust issue)

- **Affected files/functions:**
  - `src/routes/auth/forgot-password/+page.server.ts`
  - `src/lib/server/auth.ts` (`sendResetPassword`)
- **Vulnerability type:** Token exposure via untrusted URL construction
- **Risk/exploit scenario:** Manipulated host/origin can place valid reset token in attacker-controlled domain link.
- **Evidence pattern:** Reset callback derived from `url.origin` and tokenized callback parameter.
- **Recommended fix:** Build links from trusted configured base URL (for example `BETTER_AUTH_BASE_URL`) and enforce callback allowlist.
- **Verification steps:**
  1. Send reset request with spoofed host/origin.
  2. Inspect email link destination.
  3. Confirm links always target trusted canonical domain.

### [P2] Cron auth can fail open when `CRON_SECRET` is unset

- **Affected files/functions:**
  - `src/routes/api/cron/email-weekly-summary/+server.ts`
  - `src/routes/api/cron/reset-recurring-paid/+server.ts`
- **Vulnerability type:** Auth misconfiguration / fail-open
- **Risk/exploit scenario:** If `CRON_SECRET` is undefined, bearer comparison may erroneously permit `Bearer undefined`.
- **Evidence pattern:** Authorization check compares against interpolated env value without mandatory non-empty secret assertion.
- **Recommended fix:** Validate secret presence at startup and fail closed when missing.
- **Verification steps:**
  1. Run environment without `CRON_SECRET`.
  2. Call cron endpoint with crafted Authorization header.
  3. Confirm endpoint remains unauthorized/disabled.

### [P3] CSP contains permissive directives (`unsafe-inline`, `unsafe-eval`)

- **Affected files/functions:** `src/hooks.server.ts`
- **Vulnerability type:** Defense-in-depth weakness
- **Risk/exploit scenario:** If XSS is introduced elsewhere, CSP provides weaker containment.
- **Evidence pattern:** `script-src` and `style-src` include unsafe directives.
- **Recommended fix:** Migrate to nonce/hash-based CSP, reduce unsafe directives, and narrow source allowlists.
- **Verification steps:**
  1. Enable stricter CSP policy in staging.
  2. Validate app functionality with CSP reports.
  3. Remove remaining unsafe directives where feasible.

## Recommended Fixes

1. Enforce role checks in each admin action handler, not only parent layout load.
2. Generate password-reset links from a trusted configured origin and callback allowlist.
3. Require non-empty cron secret at startup and reject requests when missing.
4. Move CSP toward nonce/hash policy and remove unsafe directives where feasible.

## Residual Risks / Follow-ups

- If the product model ever changes from shared-data to user-isolated data, cross-user read/write behavior must be reclassified as a high-severity tenant-isolation vulnerability.
- Add regression tests for:

  - Password reset origin handling
  - Cron endpoint fail-closed behavior when secrets are absent
